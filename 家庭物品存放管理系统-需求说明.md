## 1. 需求说明

### 1.0 相关背景与核心痛点

- **背景：** 家庭物品放置杂乱，分布在各类柜子、榻榻米、床底等隐蔽空间。经常出现“寻找某件物品耗时过长”、“忘记物品具体存放位置”以及“特殊设备长期未充电导致电池损坏”等生活痛点。
    
- **多成员协同：** 家庭成员间信息不互通，存在“A放的东西B找不到”的情况。
    
- **核心解决方案：**  
    通过“空间拓扑数字化 + 拍照多模态识别 + 混合智能检索 + 周期性事件提醒”实现家庭资产的精细化盘点。
    
- **多端策略：**  
    核心业务逻辑（图像识别、向量计算、定时提醒）完全下沉到后端。  
    前端首期采用 **Android 客户端**（基于手机拍照的便利性），同时兼顾 **Web 端** 的日常查看与管理。
    

---

### 1.1 空间拓扑与家庭平面图绘制

系统建立**四级空间拓扑模型**：  
`地点 (Location)` → `分区 (Zone)` → `储物模块 (Container)` → `细分层级/抽屉 (Slot)`

#### 1.1.1 多地点管理

- 支持创建和切换多个独立地点（如：主住宅、车库、老家、杂物间）
    
- **缺省地点**：首次登录自动创建“我的家”
    

#### 1.1.2 动态模板快速初始化

- 提供标准户型模板（客厅、餐厅、主卧、客卧、次卧、南阳台、北阳台、主卫、次卫、厨房、电梯间）
    
- 用户可基于模板**一键生成**标准化空间拓扑，并支持自由添加、修改或删除分区
    

#### 1.1.3 储物模块定义

- 在各分区下添加对应的储物模块（如：大衣柜、五斗柜、榻榻米、床底储物箱、书桌）
    
- **多层级支持**：储物模块必须支持定义内部层级  
    （例如：主卧大衣柜 → 第一层抽屉 / 左侧挂衣区 / 顶部格子）
    

#### 1.1.4 平面图视图（一期后置）

- 支持导入 CAD 导出的 JPG/PNG 平面图
    
- 预留 2D 绘图交互接口，后续支持用户在平面图上进行“多边形打点锚定”划分分区，并将储物模块作为 Icon 拖拽摆放
    

---

### 1.2 照片上传与多模态识别入口

#### 1.2.1 核心交互流程

用户在 App 端选择特定 `Slot`（储物抽屉/层级） → 拍照或上传照片 → 触发后端 AI 异步处理 → 弹出识别结果供用户二次确认

#### 1.2.2 AI 自动化盘点（视觉驱动）

- **目标检测**：后端自动在照片上画出所有识别到的物品框（Bounding Box），用户点击框即可确认或修改标签
    
- **切片图生成**：根据识别到的物品框，自动裁剪出单个物品的微缩图（Thumbnail），用于搜索结果展示
    

#### 1.2.3 版本管理（增量更新）

- 允许用户在不删除旧照片的情况下，上传新照片覆盖/更新该储物空间的状态
    
- **旧照片自动转为“历史快照”**，支持时间轴回溯
    

#### 1.2.4 数据存储策略

- 照片原件与切片图存储在服务器指定位置（默认本地文件存储，支持一键切换至 S3 / MinIO）
    
- 提取出的物品文本标签和图片特征向量，分别写入对应数据库
    

---

### 1.3 检索与特殊功能扩展

#### 1.3.1 混合检索（Hybrid Search）

- **文本搜索**：支持精确文本、品牌名（如“罗技鼠标”）或分类标签检索
    
- **多模态搜索**：支持以图搜图，或输入模糊语义（如输入“保暖穿的”，能自动匹配到“羽绒服”的照片）
    

#### 1.3.2 搜索结果层级面包屑展示

搜索结果必须完整展示其空间链路。  
**示例**：  
搜索“发票” → 返回：[物品切片图] + 链路  
`主住宅 → 书房 → 黑色书桌 → 第二层抽屉`

#### 1.3.3 特殊设备生命周期管理（定时提醒）

**① 定期充电提醒**

- 将物品标记为“需充电设备”
    
- 默认规则：用户点击“已充电/消除提醒”后，**顺延 3 个月**自动触发下一次提醒
    
- 支持自定义周期（1 个月、半年等）
    

**② 位置状态追踪（借出/临时取出提醒）**

- 物品标记为“借出”或“临时取出”后
    
- 系统在设定时间（默认 **24 小时**）后自动推送提醒：  
    “该物品是否已归位？请及时拍照更新位置”
    
- 用户确认归位 → 自动更新位置状态并清除提醒

## 2. 技术实现参考

### 2.0 整体技术架构选型

|模块|选型|
|---|---|
|前端 / Android|Flutter|
|后端|Python 3.12 + FastAPI|
|主数据库|PostgreSQL（JSONB 支持）|
|向量数据库|Milvus 或 Qdrant|
|文本搜索引擎|Meilisearch 或 Typesense|
|异步任务 & 定时器|Celery + Redis + PostgreSQL（持久化）|
|对象存储|MinIO / AWS S3 / 阿里云 OSS|
|OCR|PaddleOCR|
|多模态标签生成|DeepSeek-V4 (Vision) 或 MiniCPM-V|

### 2.1 核心数据流向与分工

text

[用户上传照片]
    ├──> 存储服务 (S3/本地)：原图 + 压缩图 + 切片图
    └──> Celery 异步任务
           ├──> CLIP 模型 → 图片向量 → Milvus
           ├──> PaddleOCR → 图片文字 ──┐
           └──> DeepSeek-V4 → 物品标签 ─┴─> Meilisearch
                                              │
                                              ▼
                                    写入 PostgreSQL 主库

### 2.2 检索融合排序（RRF 算法）

- 同时向 `Meilisearch`（文本）和 `Milvus`（向量）发起请求
    
- 对返回结果进行加权融合排序，确保检索准确率

## 3. 核心 API 接口定义（完善版）

> 所有接口统一前缀：`/api/v1`

### 3.1 空间管理模块 (`/space`)

|方法|路径|说明|
|---|---|---|
|POST|`/locations`|创建地点|
|GET|`/locations`|获取所有地点列表|
|POST|`/zones`|在指定地点下创建分区|
|POST|`/containers`|在分区下创建储物模块|
|POST|`/containers/{container_id}/slots`|在模块下创建抽屉/层级|
|PUT|`/slots/{slot_id}`|更新层级信息|
|DELETE|`/slots/{slot_id}`|删除层级（级联提醒）|

**请求示例（创建储物模块 + 层级）**：

json

POST /containers
{
  "zone_id": "zone_客厅_001",
  "name": "五斗柜",
  "slots": [
    { "name": "第一层抽屉", "level": 1 },
    { "name": "第二层抽屉", "level": 2 }
  ]
}

---

### 3.2 物品与图像处理模块 (`/items`)

|方法|路径|说明|
|---|---|---|
|POST|`/upload`|上传照片，绑定 `slot_id`，返回 `task_id`|
|GET|`/task-status/{task_id}`|轮询 AI 识别进度|
|PUT|`/confirm/{item_id}`|用户确认/修改 AI 识别结果（标签、边界框）|
|GET|`/history/{slot_id}`|查看某储物层级的**历史快照**（照片 + 物品列表）|
|POST|`/version/rollback`|回滚到指定历史快照|

**上传响应示例**：

json

{
  "task_id": "celery_xyz123",
  "status": "processing",
  "slot_id": "slot_五斗柜_02"
}

**确认识别结果示例**：

json

PUT /confirm/item_abc123
{
  "confirmed_label": "罗技MX Master 3鼠标",
  "bounding_box": {"x": 100, "y": 150, "w": 80, "h": 60},
  "is_chargeable_device": true,
  "charge_reminder_cycle_days": 90
}

---

### 3.3 检索模块 (`/search`)

|方法|路径|说明|
|---|---|---|
|POST|`/hybrid`|混合检索（文本 / 图片 / 语义）|

**请求体**：

json

{
  "text": "保暖穿的",
  "image_file": null,
  "location_id": "loc_主住宅",
  "limit": 20
}

**返回体（含面包屑）**：

json

{
  "results": [
    {
      "thumbnail_url": "/cuts/羽绒服_thumb.jpg",
      "item_label": "羽绒服",
      "breadcrumb": "主住宅 / 主卧 / 大衣柜 / 右侧挂衣区",
      "slot_id": "slot_xxx",
      "last_updated": "2025-03-01"
    }
  ]
}

---

### 3.4 定时提醒模块 (`/reminders`)

|方法|路径|说明|
|---|---|---|
|POST|`/charge/complete`|用户确认已充电 → 顺延下一次提醒|
|PUT|`/charge/cycle`|修改提醒周期（针对特定物品）|
|POST|`/borrow`|标记物品为“借出/取出”，启动归位提醒|
|POST|`/borrow/return`|确认归位，取消归位提醒|
|GET|`/pending`|获取所有待处理的提醒列表|

#### 充电提醒顺延逻辑

json

POST /reminders/charge/complete
{
  "item_id": "item_罗技鼠标_001",
  "next_reminder_days": 90   // 默认90，可传30/180等
}

#### 借出/归位提醒

json

POST /reminders/borrow
{
  "item_id": "item_电钻_002",
  "expected_return_hours": 24,
  "borrower": "邻居老王"
}

> 系统在 `expected_return_hours` 小时后自动推送：“电钻 是否已归位？”

## 4. 需要完善或明确的设计点（补充说明）

| 维度           | 补充说明                                           |
| ------------ | ---------------------------------------------- |
| **并发与异步一致性** | 使用 `Celery` 的 `task_id` + 数据库状态字段，避免重复识别       |
| **充电提醒的持久化** | 使用 PostgreSQL 存储下一次提醒时间 + Celery Periodic Task |
| **多成员协同**    | 后续版本支持家庭共享（基本 RBAC），一期默认全家庭成员可见                |
| **安全边界**     | 照片存储路径不暴露真实路径，使用 `UUID` 重命名 + 预签名 URL          |
| **数据删除策略**   | 删除 `Slot` 时需明确：是否删除照片？保留历史？提供选项                |
| **多地点切换性能**  | 所有检索接口必须带 `location_id`，默认使用当前活动地点             |
